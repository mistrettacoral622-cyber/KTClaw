# OpenClaw Desktop Control Plane PRD
> 相比 ClawX 的增强版项目文档（Markdown 版）

## 1. 文档概述

### 1.1 项目名称
**OpenClaw Desktop Control Plane**

### 1.2 项目定位
面向 OpenClaw / ClawdBot / Moltbot 生态，打造一个比 ClawX 更完整的 **桌面 GUI + Web GUI + 本地控制台 + 多通道执行入口 + Agent Control Plane**。

目标不是只做一个聊天窗口，而是做成一个真正可用的：
- 本地智能体控制台
- 多 Agent 协同工作台
- 任务与定时任务管理面板
- 记忆/技能/MCP/通道统一治理平台
- 可远程触达的“数字员工”操作中枢

### 1.3 对标对象
本项目严格对标以下类型产品，而非泛 CLI Agent 框架：
- 桌面 GUI 客户端
- Web GUI / 本地打包 Web 客户端
- 浏览器内沙箱客户端
- 大厂深度集成客户端 / 托管控制台

### 1.4 口径说明
本文中的“Delta / 提升点”定义为：

> **竞品公开宣称，且当前 ClawX 官网、README、公开物料中未见同等明确表述的能力。**

说明：
- 这不是对未公开代码做“绝对没有”的证明。
- 纯 CLI / Agent Framework 不纳入主比较对象。
- 所有结论以当前公开材料为准。

## 2. ClawX 当前公开基线

结合当前公开资料，ClawX 的能力基线可以概括为：

- Electron + React 桌面客户端
- 内嵌 OpenClaw runtime
- 引导式安装
- Provider 可视化配置面板
- 内置 Skill Marketplace
- GUI 页面包括：Chat / Channels / Skills / Cron / Settings
- 面向 24/7 autonomous monitoring / research
- 支持 10+ AI providers
- 支持 20+ communication channels
- 兼容 MCP

### 2.1 ClawX 当前优势
- 已经有一套较完整的桌面壳
- 有基础 provider 配置能力
- 有 skill marketplace
- 有 cron / channels 等管理入口
- 已具备开发者友好的 GUI 心智

### 2.2 ClawX 当前公开短板
相较竞品，ClawX 当前公开资料里还没有被强表达或产品化的方向包括：

- 多 Agent 编排与虚拟团队
- 显式的 approvals / sandbox / permission control plane
- 更强的 IM 原生入口能力（微信 / 飞书 / 企业微信）
- 更完整的长期记忆治理能力
- 多形态部署（Tauri / 浏览器内沙箱 / 云托管）
- 配置迁移、审计、健康检查、回滚
- 多模型统一治理层与统一 MCP 治理
- 更强的任务看板 / mission control 视角

## 3. 产品目标

## 3.1 核心目标
把当前“桌面聊天入口”升级为“**Agent 控制平面**”。

具体目标：
1. **降低部署摩擦**：安装即用、本地即起、支持迁移和恢复
2. **强化控制能力**：任务、定时任务、记忆、技能、MCP、通道统一管理
3. **强化 Agent 组织能力**：支持子智能体、角色分工、并行执行、群聊发言
4. **强化远程触达能力**：飞书、钉钉、企业微信、Telegram 等通道接入
5. **强化可观测性**：会话、任务、日志、心跳、用量、执行状态统一可视化
6. **强化长期运行能力**：从聊天 GUI 升级为可长期在线的数字员工控制台

## 3.2 一句话价值主张
**不是再做一个聊天壳，而是做一个可以管理、调度、观察、治理多智能体系统的桌面控制中台。**

## 3.3 目标用户
| 用户类型 | 典型诉求 |
|---|---|
| 个人高频用户 | 希望把本地 OpenClaw 真正当作“数字管家”使用 |
| 中国办公场景用户 | 主要通过飞书、企业微信、钉钉、QQ 等入口发起和接收任务 |
| 小团队或工作室 | 需要多智能体协作、群聊触发、定时任务与可视化治理能力 |
| 进阶开发者 | 需要对技能、工具、MCP 服务和运行权限进行显式管理 |

## 3.4 非目标
- 第一阶段不自研新的大模型推理内核
- 第一阶段不替代 OpenClaw runtime 的核心能力
- 第一阶段不直接做重型企业 SaaS 平台，而是优先把本地端和轻量远程控制打实

## 3.5 核心产品原则
1. **默认可用**：安装后先能跑，再通过引导逐步打开高阶能力

2. **显式治理**：技能、工具、MCP、记忆、文件、子智能体都必须可见、可查、可控

3. **中国场景优先**：飞书、企业微信、钉钉、QQ 等通道应按一等入口设计

4. **多智能体产品化**：框架、分身、群聊协作不能只靠手写提示词实现

5. **本地优先，远程可达**：本地数据默认可留存，同时保留 Web 或远程入口能力

## 6.2 核心对象模型建议
为避免后续功能扩展继续依赖隐式配置，建议在实现层面显式抽象出以下核心对象：

- `FrameworkTemplate`：框架模板定义，描述组织结构、默认角色集合和协作方式
- `AgentRole`：角色定义，约束子智能体的职责边界、默认技能和行为风格
- `ExecutionPolicy`：执行策略，定义并行度、审批要求、回流策略和失败处理方式
- `MemoryPolicy`：记忆策略，定义记忆的继承、共享范围、写入权限和保留周期
- `ToolPolicy`：工具策略，定义不同 Agent、框架和通道可访问的工具范围与风险级别

这一层对象模型的价值在于，把“框架协作”“权限控制”“记忆治理”从零散开关提升为可配置、可复用、可导入导出的结构化能力。

## 9.4 远程 Web 控制台
建议把“桌面端本机运行，浏览器轻量治理”作为明确差异化方向，而不是附属功能。

目标：

- 让用户离开当前电脑后，仍能查看运行状态、处理审批并追踪任务执行

核心能力：

- 查看 Dashboard、Cron、Agent、Memory、Files 等关键面板
- 批准或拒绝高风险动作
- 远程查看最近执行状态、失败原因和恢复建议
- 作为桌面客户端的轻量 companion，而不是独立替代品

## 9.5 审批与权限模式
建议把高风险操作从“隐式执行”升级为“显式治理”，并提供清晰的权限模式分层：

- `Conservative`：默认只读，不允许命令执行和写文件
- `Standard`：允许常规工具，危险操作必须二次确认
- `Full`：面向高级用户，开放完整能力
- `Custom`：按工具类别、Agent、框架或通道自定义权限

这部分能力补上后，产品才能真正从“能运行”提升到“可治理、可审计、可放心交付”。

## 9.6 预置工作流与模板包
为了降低非技术用户的上手成本，建议将“模板”从简单角色预设升级为可直接落地的工作流模板包。

建议预置模板：

- 办公助理
- 信息监控
- 搜索研究
- 群聊秘书
- 定时报告
- 客服跟进

模板包建议同时包含：

- 默认 Agent 组合
- 推荐 Skills
- 推荐 Tools
- 推荐 Channels
- 推荐 Cron 配置
- 默认记忆策略

这样可以把“安装完成”进一步推进到“开箱就能进入工作状态”。

## 12. 风险与注意事项

## 12.1 产品风险
- 功能过多导致复杂度上升
- 新用户可能被“控制台化”界面吓退
- 多 Agent 协同容易造成认知负担
- 记忆可视化后，用户对“为什么写入”会更敏感

## 12.2 技术风险
- 多通道接入维护成本高
- 本地工具权限与安全策略复杂
- 定时任务、监听、心跳引入后台常驻成本
- 文件、记忆、Agent 状态的一致性需要单独设计

## 12.3 建议策略
- 默认提供“简单模式”和“专业模式”
- V1 先做可见可控，不一次性上全部治理逻辑
- 功能入口遵循“先展示，后开启”
- 子智能体和框架用模板化方式降低复杂度

## 13. 最终结论

OpenClaw 桌面端下一阶段不应该只是做一个更漂亮的 ClawX，而应该做成：

> **一个能管理 Agent、管理记忆、管理技能、管理通道、管理任务、管理运行状态的桌面控制中台。**

真正的差异化不在“再多一个聊天框”，而在于：

1. **从聊天入口升级为控制平面**
2. **从单 Agent 升级为多 Agent 组织系统**
3. **从隐式运行升级为可观测、可治理、可迁移**
4. **从开发者 GUI 升级为面向真实办公与长期值守的数字员工平台**
